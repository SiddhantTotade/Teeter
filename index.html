Teeter Documentation

A high-performance Proxying Load Balancer built with Go. Designed to be extremely fast, infinitely resilient, and deeply observable. Whether you're routing web traffic or internal microservices, Teeter is built to withstand chaos.

Load Balancing 101 (For Beginners)

Imagine a very popular restaurant with just one waiter. When a hundred customers arrive, that waiter will become overwhelmed, make mistakes, and eventually collapse. The solution? Hire ten waiters and place a "Host" at the front door to direct each incoming customer to the least busy waiter.

In web infrastructure, your application servers are the waiters, and a Load Balancer (like Teeter) is the Host. It sits directly in front of your servers, intercepts incoming network traffic, and intelligently distributes it across your raw servers.

1. Routing (Prefixing)

Load balancers inspect the requested URL. Just like a mail sorting room, if a user requests /api/login, the load balancer recognizes the "api" part and routes the user strictly to the backend servers dedicated strictly to API tasks.

2. Health Checking

What if one of the backend servers suddenly crashes? A smart load balancer constantly "pings" every server in the background (asking "Are you alive?"). If a server fails to respond, the load balancer temporarily removes it from the list of healthy destinations.

3. Balancing Strategies

How does the load balancer know which alive server to send traffic to?

  • Round Robin: Handing requests out eagerly in a circle. Server A gets Request 1, B gets 2, C gets 3, then back to A.
  • Least Connections: Sending the request to whichever server is currently working on the fewest active things.
  • Weighted: Directing 80% of traffic to a powerful server, and 20% to a weaker one.

How Teeter specifically works

Teeter combines the basics of load balancing with modern cloud-native resiliency patterns.

[ User Request ] │ ▼ ( Gateway: Match URL path ) │ ▼ [ Queue: Prevent sudden spikes ] │ ▼ ( Rate Limit: Stop spam ) │ ▼ ( Strategy: Pick a backend ) │ ▼ [ Circuit Breaker: Fail fast on errors ] │ ▼ ( Final Delivery to Target Server )

Special Mechanisms in Teeter:

  • Circuit Breaking: If a specific backend starts throwing HTTP 500 errors continuously, Teeter "breaks" the circuit, stopping all traffic to it for a cooldown period to give it a chance to recover.
  • Retry Logic: If a connection fails mid-flight, Teeter invisibly tries a different healthy server before returning an error to the user, saving the user experience.
  • Worker Pools: Instead of spinning up an infinite amount of invisible background tasks (Goroutines) and crashing itself by running out of memory, Teeter restricts itself to a specific limit of concurrent workers to guarantee low-latency.

Complete File Tour

Every single line of code in Teeter belongs to a well-defined domain. Below is the precise meaning of every file in the repository.

Core Entry & Configuration

lb/cmd/lb/main.go

The main entry point. Reads the config, initializes every component (routers, strategies, health), and starts the primary HTTP server loop.

lb/pkg/config/config.go

Parses the config.yaml file into a Go struct so the application understands your routing rules and timeout preferences.

config.yaml

Your user-facing brain. It's where you define the routes, algorithms, and backend URLs.

The Traffic Logic (lb/internal/)

gateway/gateway.go

Accepts raw HTTP requests and funnels them into the load balancer lifecycle.

gateway/router.go & route.go

Inspects the request URL string and matches it against your configured Prefix filters.

registry/registry.go

A thread-safe memory map keeping strict track of which configured backend servers are currently alive vs. dead.

health/checker.go

Runs continuous background timers to ping every backend's health-check URL, automatically updating the registry if one goes down.

handler/handler.go

Generates the Reverse Proxy logic. It actually performs the act of taking user data and forwarding it securely to a target server.

handler/error_pages.go

If every single backend server is completely offline, this returns a beautiful "Service Unavailable" HTML page instead of an ugly raw text error.

Resiliency & Math (lb/internal/)

queue/queue.go

If Teeter receives 10,000 requests in a split second, the Queue briefly buffers them so backend servers don't crash from the flood.

retry/retry.go

Seamlessly recovers dropped requests transparently by routing to the next healthiest alternative.

circuitbreaker/circuitbreaker.go

Enforces maximum error thresholds, "tripping" and stopping all flow to failing domains for safety.

ratelimit/bucket.go

Limits the number of requests a single IP address can originate per minute, preventing DoS attacks.

Selection Strategies (lb/internal/strategy/)

strategy.go

The generic Interface defining how algorithms must be built.

roundrobin.go

Rotates sequentially through every active backend linearly.

leastconnections.go

Maintains a counter of active requests. Routes the next request to the server with the lowest current count.

weightedroundrobin.go

Allocates requests based on an assigned integer weight (e.g. powerful server receives 5x the requests of a normal server).

../consistenthashing/...

Uses a cryptographic hash of the User's IP so that a specific user ALWAYS gets routed to the same backend server (useful for session stability).

Monitoring & DevOps

admin/admin.go

Spins up a separated private port (usually `:1997`) that returns JSON stats on the load balancer's current performance.

metrics/metrics.go

Records latency distributions and error codes format-ready for Prometheus scraping.

monitoring/...

Contains prometheus.yml and grafana/datasources which pre-configure external tools to visualize Teeter's statistics on gorgeous charts automatically.

Dockerfile & docker-compose.yml

Container tooling. Compiles everything into a tiny binary image and launches it immediately alongside its monitoring databases.

General Example Scenario

To put this together, imagine you are building an online Shoe Store API. Thousands of users are trying to buy shoes at the same time.

The Setup

Because there is so much traffic, you run three identical cloud server instances of your API. Let's assume you launch Teeter strictly to sit in front of these instances.

# config.yaml (You write this once)
routes:
  - prefix: "/api/"
    strategy: "round_robin"
    backends:
      - url: "http://shoes-server-1:8080"
      - url: "http://shoes-server-2:8080"
      - url: "http://shoes-server-3:8080"

The Lifecycle of a Request

  1. A user opens their browser and clicks "Purchase". The browser sends a request to http://your-domain.com/api/buy.
  2. Teeter's Gateway (gateway.go) receives the request before any of your servers ever see it.
  3. The Router (router.go) checks the config.yaml and sees that /api/... is explicitly defined.
  4. The Registry (registry.go) tells the load balancer that all three shoe servers are currently ALIVE and passing their health checks.
  5. The Strategy (roundrobin.go) decides that because Server #1 was used last time, Server #2 should be used this time.
  6. The request is seamlessly forwarded straight to shoes-server-2. The user's shoe disappears from inventory, and Server #2 replies back through Teeter.
  7. Teeter's Metrics engine logs the duration of the entire interaction, making it visible to your DevOps team via Grafana.

Getting Started

Deploy Teeter in seconds using Docker. The ecosystem comes pre-configured for instant setup.

1. Start the Environment

Spin up the Gateway, Backend, and Prometheus natively.

docker-compose up -d

2. Verify Logs

Ensure all traffic processes boot successfully.

docker logs -f teeter

3. Access Dashboard

  • Admin API Health: http://localhost:1997/health
  • Live Statistics (Grafana): http://localhost:3000
